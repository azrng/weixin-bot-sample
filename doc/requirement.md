# 微信官方 iLink Bot API 演示 Demo 需求文档

## 1. 项目背景

当前仓库用于编写一个微信官方 iLink Bot API 的演示 Demo。这个 Demo 的目标不是做完整后台系统，而是做一个可以现场演示的最小可用示例，让观众看到：

- 如何绑定微信 Bot
- 用户给 Bot 发消息后，系统如何自动回复
- 系统如何主动给用户推送一条消息

用户指定可以参考 `C:\Work\github\OpenCowork` 中现有的微信推送渠道代码，因此本项目优先复用已经验证过的协议处理经验，而不是重新摸索接口细节。

## 2. 目标与范围

### 2.1 目标

- 提供一个基于 Blazor Server 的微信官方 iLink Bot API 演示 Demo。
- Demo 页面直接展示微信绑定、主动推送、入站消息与自动回复结果。
- 用户在微信里给 Bot 发消息后，Bot 能返回固定祝福语。
- 页面刷新后仍能看到最近配置、绑定状态、消息记录和推送记录。
- Demo 协议细节尽可能贴近 `doc/iLinkBot协议.md`，包括标准鉴权头、连接自检、`get_updates_buf` 游标持久化、`context_token` 缓存与 `typing_ticket` 使用。

### 2.2 一期范围

- 应用图标与应用名称
- iLink Bot 基础配置管理

- 绑定状态展示
- 启动监听与停止监听
- 连接自检与 Typing Ticket 获取
- 长轮询接收微信文本消息
- 固定祝福语自动回复
- 主动推送消息示例按钮
- 已知联系人与上下文 token 缓存展示
- 最近消息记录与操作日志展示
- 基于本地文件的真实持久化

### 2.3 非本期范围

- 登录页
- 权限系统
- 多用户管理
- 多渠道接入
- 图片、语音、文件等多媒体消息处理
- AI 大模型回复
- 外部数据库与复杂部署方案

## 3. 用户角色

### 3.1 演示操作员

## 5. 核心业务流程

### 5.1 微信绑定流程

1. 演示操作员打开 Demo 页面。
2. 填写或确认 iLink Bot 基础配置。
3. 点击“绑定微信”或“生成二维码”。
4. 系统调用 iLink API 获取二维码并展示。
5. 用户使用微信扫码。
6. 系统轮询扫码状态。
7. 绑定成功后保存 Token、账号 ID、用户 ID、绑定账号名和绑定时间。
8. 页面刷新后仍显示已绑定状态。

### 5.2 用户发消息自动回复流程

1. 演示操作员点击“启动监听”。
2. 系统长轮询 `getupdates` 获取微信用户消息。
3. 当收到文本消息时，系统将消息保存到本地记录。
4. 系统按固定模板生成祝福语，例如“祝您今天顺顺利利，万事如意。”
5. 系统调用 `sendmessage` 将祝福语回发给微信用户。
6. 页面展示最近收到的消息和最近一次回复结果。

### 5.3 主动推送流程

1. 演示操作员在主动推送区填写目标 `ExternalChatId`、`ContextToken` 和消息内容。
2. 点击“推送演示消息”。
3. 系统优先调用 `getconfig` 获取 `typing_ticket`，并尝试发送 `sendtyping` 状态。
4. 系统调用 iLink `sendmessage` 接口向指定用户或会话发送消息。
5. 页面展示发送结果、响应摘要和发送时间。
6. 推送记录写入本地日志并在页面可见。

### 5.4 协议自检与上下文续聊流程

1. 演示操作员点击“验证连接”。
2. 系统调用 `getconfig` 校验当前 `bot_token` 是否仍可用，并获取 `typing_ticket`。
3. 页面展示最近一次自检结果、Typing Ticket 与更新时间。
4. 当系统收到用户消息时，会将 `ExternalChatId`、`SenderName`、`context_token` 与最近消息摘要写入已知联系人缓存。
5. 操作员可直接从已知联系人列表带入推送目标，在应用重启后继续与已联系用户对话。

## 7. 可见范围

- Demo 对外只提供一个演示页面。
- 不实现登录和权限控制。
- 所有操作入口都集中在同一页面，方便演示。

## 8. 边界场景与异常

- 未配置基础参数时禁止启动监听，并给出明确提示。
- 二维码过期时允许刷新重新生成。
- 绑定轮询超时时给出失败说明，并保留重试入口。
- `getupdates` 请求超时时，系统自动重试并记录日志。
- 长轮询连续失败 3 次后进入 30 秒退避重试，避免持续高频请求。
- 消息没有 `context_token` 时，不执行回包并记录原因。
- `sendmessage` 失败时，页面展示错误详情且保留该条操作记录。
- 若接口返回 `errcode = -14`，系统标记当前会话已过期，要求重新扫码绑定。
- 页面加载不到配置或日志文件时，展示错误态并允许重试。

## 9. 非功能要求

- 项目使用 UTF-8 编码。
- 页面必须有 `loading`、`empty`、`error` 状态。
- 所有按钮必须为真实可交互动作。
- 刷新浏览器后仍能看到配置、绑定状态、最近消息与日志。
- 刷新浏览器后仍能保留最近一次 `get_updates_buf` 与已知联系人的 `context_token` 缓存。
- 优先复用 `OpenCowork` 中已验证的微信协议处理逻辑，避免重新猜测协议。

## 10. 已确认约束

- 不需要登录页。
- 自动回复暂时固定为祝福语，不接 AI。
- Demo 的重点是“微信绑定成功后可以正常收发消息”和“演示主动推送”。

### 11.5 二期核心流程

#### 11.5.1 媒体上传发送流程

1. 演示操作员在页面选择目标联系人与媒体类型。
2. 系统读取本地文件，计算文件长度与 MD5。
3. 系统生成一次性 AES-128 key。
4. 系统调用 `getuploadurl` 获取上传参数。
5. 系统使用 AES-128-ECB + PKCS7 对文件内容加密。
6. 系统把密文上传到 `https://novac2c.cdn.weixin.qq.com/c2c/upload`。
7. 系统从响应头中提取下载参数。
8. 系统调用 `sendmessage` 发送对应媒体消息结构。
9. 页面展示上传、发送结果与关键元信息。

#### 11.5.2 媒体下载回读流程

1. 长轮询收到媒体消息后，系统记录媒体元数据。
2. 演示操作员点击“下载并解密”。
3. 系统调用 CDN 下载地址获取密文内容。
4. 系统使用消息携带的 AES key 进行解密。
5. 解密后的文件写入本地缓存目录。
6. 页面展示下载结果、缓存路径与失败原因。

#### 11.5.3 Typing 状态与体验说明流程

1. 演示操作员点击“验证连接”或执行主动发送。
2. 系统通过 `getconfig` 获取 `typing_ticket`。
3. 页面展示“正在输入”能力说明，解释其作用是提升人机交互体验。
4. 在真实发送消息前，系统先尝试调用 `sendtyping`。
5. 页面记录该步骤是否成功，并展示失败时的原因说明。

#### 11.5.4 语音消息接收与 ASR 展示流程

1. 微信用户发送语音消息。
2. 系统通过 `getupdates` 收到语音消息元数据。
3. 页面展示语音时长、编码格式、媒体标识和协议自带的 ASR 文本。
4. 操作员可选择下载并解密语音内容。
5. 若 ASR 文本缺失，页面明确提示“协议未返回转写文本”，但保留媒体下载能力。

#### 11.5.5 真实账号联调流程

1. 确认可用的 `bot_token`、`accountId`、`userId`、可扫码微信账号与目标联系人。
2. 执行二维码绑定与连接自检。
3. 执行 `sendtyping` 和主动推送体验验证。
4. 执行文本收消息、自动回复验证。
5. 执行图片/语音/文件/视频至少各一条发送或接收验证。
6. 对收到的媒体消息执行下载并解密验证。
7. 执行 `errcode=-14` 会话过期与重新扫码恢复验证。
8. 记录结果、失败原因和需要重试的步骤。

### 11.7 二期边界与异常

- 本地文件不存在或无法读取时，阻止上传并展示明确错误。
- `getuploadurl` 失败时，不执行后续加密上传。
- CDN 上传成功但 `sendmessage` 失败时，保留上传结果和失败日志，方便复用同一媒体参数重发。
- 下载解密失败时，保留原始媒体元数据与失败原因，不覆盖已有缓存文件。
- 若语音消息缺少 ASR 文本，不阻断下载，只标记“无转写结果”。
- 若真实联调账号会话过期，checklist 当前步骤标记失败，并要求重新扫码绑定。
- 若协议未公开速率限制，页面以说明文案提示“能力已接入，但上线前需做压测与限流保护”。
- 若群聊相关字段出现但能力边界不明确，页面仅做原始字段展示与风险说明，不承诺群聊流程打通。
- 历史消息不可回溯时，页面明确提示“仅支持游标向前消费，不支持补拉历史消息”。

### 11.8 二期非功能要求

- 媒体上传下载过程必须有明确的阶段状态：准备中、加密中、上传中、发送中、成功、失败。
- 本地缓存文件可追踪、可清理，避免无限增长。
- 真实联调 checklist 支持刷新页面后保留最近结果。
- 页面中的解释性文本要尽量口语化，让非研发观众也能看懂每个能力在演示什么。
