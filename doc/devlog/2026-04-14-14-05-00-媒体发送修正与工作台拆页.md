# 本次目标

- 继续排查媒体消息真实联调失败问题，并把过长的单页演示台改成左右结构的多页面工作台。

# 核心改动

- 调整媒体发送链路：`getuploadurl` 之后发送的媒体结构改为优先使用 AES 加密后的密文长度，并同时补发 `aeskey` 与 `aes_key` 两种字段名，降低协议字段差异导致的失败概率。
- 为媒体记录补充密文大小字段，页面中可同时查看明文大小和密文大小，方便联调比对。
- 新增工作台式左侧导航，按“总览 / 消息 / 媒体 / 联调”拆分为多个 Razor 页面，避免首页过长。
- 抽出共享页面基类与共享工作台壳组件，统一浮层提示、顶部状态、左侧导航和页面头部结构。
- 新增媒体链路 Trace 落盘，发送媒体时会把 `getuploadurl`、CDN 上传和 `sendmessage` 的关键请求/响应写到 `App_Data/media-trace/`，便于真实联调排查。
