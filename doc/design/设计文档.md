# 项目设计文档

## 1. 微信官方 iLink Bot API 演示 Demo

### 1.1 架构设计

- 技术栈采用 `.NET 10 + Blazor Server + BootstrapBlazor`，以单项目方式同时承载 UI、服务层与本地持久化，优先保证 Demo 可快速落地和稳定演示。
- 项目采用轻量分层：
  - `Components/Pages` 负责演示页面与交互
  - `Services` 负责微信 API 调用、消息监听、主动推送和固定回复编排
  - `Models` 负责配置、绑定会话、消息记录、日志 DTO
  - `Storage` 负责 JSON 文件持久化
- 微信协议处理优先复用 `C:\Work\github\OpenCowork` 中官方微信渠道服务的成熟实现思路，并裁剪为本 Demo 的核心能力：
  - 获取二维码 `get_bot_qrcode`
  - 轮询扫码结果
  - 长轮询获取消息 `getupdates`
  - 获取配置与 `typing_ticket` 的 `getconfig`
  - 发送“正在输入”状态 `sendtyping`
  - 发送文本消息 `sendmessage`
- 协议状态做真实持久化，保证应用重启后仍然可继续演示：
  - `get_updates_buf` 作为长轮询游标写入本地状态
  - 最近联系人 `context_token` 缓存写入本地状态
  - 最近一次连接自检结果和 `typing_ticket` 写入本地状态
- 本 Demo 不实现登录页和权限系统，默认直接进入演示页，减少与演示目标无关的流程。
- 为保证刷新后状态仍可见，一期采用 `App_Data` 下的 JSON 文件保存：
  - 基础配置
  - 绑定结果
  - 最近入站消息
  - 最近联系人上下文缓存
  - 最近回复记录
  - 最近主动推送记录
  - 最近连接自检结果
  - 操作日志

### 1.2 功能需求

#### 1.2.1 模块划分

- 演示框架模块
  - 应用图标与应用名称
  - 页面头部说明
- 微信 Bot 演示模块
  - 基础配置
  - 微信绑定
  - 协议自检
  - 监听控制
  - 已知联系人上下文缓存
  - 主动推送
  - 入站消息记录
  - 固定祝福语回复
  - 日志展示

#### 1.2.2 业务规则

- 页面默认直接进入 Demo 主页面。
- 微信绑定成功后，系统立即持久化 `Token`、`AccountId`、`UserId`、绑定状态、绑定时间。
- 启动监听前必须校验基础配置是否完整。
- 系统同一时刻只允许一个监听实例运行。
- 每次请求都重新生成 `X-WECHAT-UIN`，避免复用同一个随机值。
- `getupdates`、`getconfig`、`sendtyping` 请求统一带 `base_info.channel_version=1.0.3`。
- `getupdates` 返回的新 `get_updates_buf` 必须立即覆盖旧值并落盘持久化。
- 入站消息只处理文本消息；非文本消息记录为“已忽略”。
- 收到文本消息后，默认回固定祝福语。
- 收到文本消息后缓存 `ExternalChatId`、`SenderName` 和 `context_token`，供主动推送继续使用。
- 主动推送和自动回复前优先尝试获取 `typing_ticket` 并发送 `sendtyping`。
- 当接口返回 `errcode = -14` 时，系统标记会话过期并要求重新扫码绑定。
- 长轮询连续失败 3 次后进入 30 秒退避重试，不直接退出监听。
- 主动推送属于显式按钮操作，执行后必须写入结果记录。
- 页面上的“绑定微信”“刷新二维码”“启动监听”“停止监听”“推送演示消息”必须是真实有效的操作。

#### 1.2.3 固定回复规则

- 一期不接入 AI。
- 默认回复模板为固定祝福语，例如：
  - `祝您今天顺顺利利，万事如意。`
  - `感谢您的消息，祝您生活愉快、事事顺心。`
- 页面可展示当前固定回复模板，便于演示说明。

### 1.4 交互说明

#### 1.4.1 配置保存交互

- 演示操作员修改基础配置后点击“保存配置”立即落盘。
- 保存成功后更新页面摘要状态。
- 若关键字段缺失，则提示错误并阻止启动监听。

#### 1.4.2 微信绑定交互

- 点击“绑定微信”后调用二维码接口。
- 若已有未过期绑定会话，则优先复用；点击“刷新二维码”强制重新获取。
- 页面展示二维码图片、扫码说明、会话 Key 与开始时间。
- 后台轮询扫码结果，成功后自动更新绑定状态与基础配置字段。
- 若二维码过期，则界面提示“二维码已过期”，并保留刷新入口。

#### 1.4.3 监听交互

- 点击“启动监听”时，系统先校验绑定状态和关键配置。
- 启动成功后运行后台长轮询服务并写入运行时间。
- 点击“停止监听”时取消轮询并写入停止时间。
- 若轮询连续失败但可恢复，则自动按“短暂等待 → 连续 3 次失败后 30 秒退避”策略重试并写日志；若会话过期，则状态切换为错误并提示重新绑定。

#### 1.4.4 协议自检交互

- 点击“验证连接”后调用 `getconfig`。
- 若成功返回，则保存 `typing_ticket`、更新时间与最近一次自检结果。
- 若返回会话过期错误，则页面提示重新扫码绑定。
- 自检结果卡片显示最近时间、结果说明和响应摘要，便于演示协议细节。

#### 1.4.5 用户发消息自动回复交互

- 当长轮询收到文本消息时，系统生成一条入站消息记录。
- 系统同步更新已知联系人缓存，持久化 `context_token` 以支持后续续聊。
- 若存在有效 `typing_ticket`，则先发送 `sendtyping` 再回消息。
- 系统根据固定模板生成祝福语回复。
- 系统调用 `sendmessage` 回发给微信用户。
- 页面自动刷新最近消息和最近回复区域，保证演示可见。
- 若消息缺少 `context_token`，则仅记录消息和失败原因，不执行回包。

#### 1.4.6 主动推送交互

- 操作员输入 `ExternalChatId`、`ContextToken` 和消息内容。
- 操作员也可以直接从“已知联系人缓存”列表带入最近对话上下文。
- 发送前优先刷新 `typing_ticket`，并尝试发送“正在输入”状态。
- 点击“推送演示消息”后调用 `sendmessage`。
- 页面显示发送结果、响应摘要和发送时间。
- 结果写入主动推送记录和日志区，便于演示。

#### 1.4.7 边界与错误处理

- API 请求失败：展示错误提示、保留最近一次成功状态、写日志。
- JSON 文件损坏：页面进入错误态，并提供重置示例数据入口。
- 二维码轮询超时：展示失败信息并保留重新生成入口。
- `errcode = -14`：立即中止当前监听与主动推送链路，提示重新扫码绑定。
- 回包失败：保留入站消息，并在回复结果列中标记失败原因。

### 1.5 阶段边界

- 本设计文档对应阶段 0。
- 用户确认后进入阶段 1，实现 Blazor Server 项目骨架、单页 Demo、服务层、本地持久化与真实微信 API 示例链路。

## 2. 二期：媒体消息上传下载与真实账号联调

### 2.1 架构设计

- 在现有 `WeixinBotDemoService` 上继续拆分二期能力，避免把文本链路和媒体链路混在一起：
  - `WeixinBotDemoService.Media.cs`
    - 负责 `getuploadurl`
    - 负责媒体加密、上传、消息结构组装
    - 负责媒体下载、解密与本地缓存
  - `WeixinBotDemoService.Checklist.cs`
    - 负责真实账号联调 checklist 状态管理
    - 负责记录每个验证步骤的通过/失败证据
- 新增模型层对象：
  - 媒体发送请求 DTO
  - 媒体消息记录 DTO
  - 联调检查项 DTO
  - 媒体下载结果 DTO
- 文件存储延续当前 JSON + 本地目录模式：
  - `App_Data/weixin-demo-state.json` 保存媒体记录摘要与 checklist 结果
  - `App_Data/media-cache/` 保存下载解密后的文件
  - `App_Data/media-temp/` 保存待上传的加密临时文件或上传中间产物
- 协议实现以 `doc/iLinkBot协议.md` 为准，核心新增链路：
  - `POST /ilink/bot/getuploadurl`
  - `POST https://novac2c.cdn.weixin.qq.com/c2c/upload`
  - `GET https://novac2c.cdn.weixin.qq.com/c2c/download`
- 页面结构除“可操作功能区”外，增加“协议说明区”，用于展示：
  - 当前已覆盖的端点
  - 每个端点对应的用户可见场景
  - 当前支持状态、依赖项和已知限制
- 解释性文本不只是提示语，而是功能的一部分：
  - 用于辅助演示者讲解为什么先 `getconfig` 再 `sendtyping`
  - 用于解释为什么媒体上传要先加密再上 CDN
  - 用于提示历史消息、群聊、限流等能力边界
- 媒体链路的密码学边界：
  - 上传前使用随机 16 字节 AES key
  - 加密算法为 AES-128-ECB + PKCS7
  - 媒体消息中持久化 `aes_key`、`download_param` 等必要字段，以支持后续下载回读

### 2.2 功能需求

#### 2.2.1 模块划分

- 媒体消息演示模块
  - 媒体文件选择
  - 上传前摘要展示
  - 上传发送动作
  - 最近媒体记录
  - 下载并解密动作
- 协议说明模块
  - 协议能力矩阵
  - 端点与场景对照说明
  - 已知限制与风险说明
- 真实联调模块
  - 联调前置准备说明
  - 可执行 checklist
  - 每一步结果记录
  - 风险提示与重试建议

#### 2.2.2 业务规则

- 只支持已知联系人上下文发媒体消息，不做无上下文的盲发。
- 媒体发送前必须校验：
  - 已完成绑定
  - 已有可用 `context_token`
  - 已选择本地文件
- 图片、语音、文件、视频统一走“获取上传参数 → 加密 → CDN 上传 → sendmessage”的链路。
- 下载只对最近记录中的媒体消息开放，且需要消息中已有 `download_param` 与 `aes_key`。
- 真实联调 checklist 的每一步必须是可重复执行的按钮动作，而不是静态说明。
- checklist 结果需要持久化，便于刷新页面后继续联调。
- 协议说明区必须覆盖至少以下场景：
  - 二维码登录
  - 连接自检
  - Typing 状态
  - 文本消息
  - 图片消息
  - 语音消息与 ASR
  - 文件消息
  - 视频消息
  - 媒体下载回读
  - 会话过期恢复
- 对官方未明确的能力，页面统一展示为“状态待确认”或“仅做说明”，不要伪装成已完全支持。

#### 2.2.3 权限与阶段边界

- 本期仍不引入登录页和权限系统。
- 二期仍保持单页 Demo，不拆后台模块。
- 用户确认本章节设计后，再进入阶段 1 开发。

### 2.4 交互说明

#### 2.4.1 媒体发送交互

- 选择媒体类型与本地文件后，页面先展示文件名、大小、MD5 摘要。
- 点击“发送媒体消息”后，依次执行：
  - `getuploadurl`
  - AES-128-ECB 加密
  - CDN 上传
  - `sendmessage`
- 每个阶段都要写日志，并更新页面状态。
- 若中途失败，保留当前步骤的失败信息，不自动清理中间结果。

#### 2.4.2 协议说明交互

- 协议能力总览区以“能力矩阵 + 解释文案”的方式展示，不要求用户操作才可见。
- 每个能力项都要说明：
  - 它对应哪个协议端点
  - 当前 Demo 是否支持
  - 用户在页面哪个区块可以触发
  - 已知限制或风险
- 对于语音 ASR、会话过期、群聊待确认等内容，需要增加辅助说明文本，帮助演示者解释功能边界。

#### 2.4.3 媒体下载交互

- 对收到或已发送的媒体记录，提供“下载并解密”按钮。
- 点击后从 CDN 下载密文内容，再用对应 AES key 解密。
- 解密成功后，把文件写入 `App_Data/media-cache/` 并展示缓存路径。
- 若解密失败，记录日志并保留该媒体记录，允许重新下载。

#### 2.4.4 联调 checklist 交互

- checklist 中每个步骤都是独立动作，允许单独执行和重复执行。
- 每次执行后更新状态、时间、证据摘要和失败信息。
- 若依赖前置步骤未完成，则当前步骤直接提示缺失条件，不进入真实请求。
- “验证媒体链路”至少覆盖：
  - 选择文件
  - 上传
  - 发送
  - 下载
  - 解密
- “验证协议场景覆盖”至少覆盖：
  - 登录
  - getconfig
  - sendtyping
  - 文本消息
  - 图片
  - 语音 ASR
  - 文件
  - 视频
  - 会话过期恢复

#### 2.4.5 错误与风险处理

- 上传参数异常：标记“上传准备失败”，建议重新获取上传参数。
- CDN 上传失败：保留失败响应和文件摘要，避免再次计算。
- 媒体消息结构缺字段：阻止发送并给出缺失字段名称。
- 会话过期：checklist 步骤失败并引导重新扫码绑定。
- 本地缓存目录不可写：下载步骤失败，提示检查文件权限。
- 历史消息不可回溯：在协议说明区持续显示限制说明。
- 速率限制未公开：在联调 checklist 和说明区都提示“仅验证功能，不承诺高频稳定性”。
- 群聊边界不明确：相关字段只做原始展示，不把它描述为已验证完成的能力。
