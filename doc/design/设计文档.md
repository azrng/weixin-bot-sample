# 项目设计文档

## 1. 微信官方 iLink Bot API 示例

### 1.1 架构设计

- 技术栈采用 `.NET 10 + Blazor Server + BootstrapBlazor`，以单项目方式同时承载 UI、服务层与本地持久化能力，避免在空仓库里一开始就拆过多层级。
- 一期采用 `Web -> Services -> Storage` 的轻量分层：
  - `Components/Pages` 负责页面和交互
  - `Services` 负责微信 API、认证态、页面状态和业务编排
  - `Models` 负责配置、绑定会话、消息记录和日志 DTO
  - `Storage` 负责 JSON 文件读写与状态恢复
- 微信协议适配优先复用 `C:\Work\github\OpenCowork` 中官方微信渠道服务的成熟逻辑，并裁剪为示例项目所需的四个核心能力：
  - 获取二维码 `get_bot_qrcode`
  - 轮询扫码结果
  - 长轮询获取消息 `getupdates`
  - 回发文本消息 `sendmessage`
- 页面交互数据流：
  - 页面事件触发服务方法
  - 服务调用 iLink API 并更新本地状态
  - 本地状态持久化到 `App_Data` 下的 JSON 文件
  - 页面通过 Scoped 状态服务刷新展示
- 为了保证“刷新后仍生效”，一期不使用内存假数据；配置、绑定结果、运行摘要、测试推送记录、消息日志都要写入本地文件。
- 管理员默认拥有所有菜单权限，因此权限实现采用轻量登录态控制，不引入复杂 RBAC。

### 1.2 功能需求

#### 1.2.1 模块划分

- 认证模块
  - 登录页
  - 默认管理员登录
- 系统框架模块
  - 顶部应用图标与应用名称
  - 左侧双层菜单
  - 首页仪表盘
- 微信 Bot 示例模块
  - 基础配置
  - 微信绑定
  - 监听控制
  - 测试推送
  - 入站消息
  - 自动回复
  - 交互日志

#### 1.2.2 菜单设计

- 一级菜单
  - 工作台
  - 示例中心
  - 系统管理
- 二级菜单
  - 工作台 / 首页
  - 示例中心 / 微信 Bot 示例
  - 系统管理 / 登录会话

#### 1.2.3 业务规则

- 管理员登录后默认拥有全部菜单权限。
- 微信绑定成功后，系统立即持久化 `Token`、`AccountId`、`UserId`、绑定状态、绑定时间。
- 监听启动前必须校验基础配置是否完整。
- 系统同一时刻只允许一个微信监听实例运行。
- 入站消息仅处理文本消息；非文本消息记录为“已忽略”。
- 自动回复关闭时，系统仍保存入站消息，但不回发微信。
- 测试推送属于显式操作，执行后必须记录请求与结果摘要。
- 页面中所有“绑定微信”“启动监听”“停止监听”“发送测试消息”“清空日志”等按钮都必须真实生效。

### 1.4 交互说明

#### 1.4.1 登录交互

- 访问根路由时，未登录用户跳转到登录页。
- 登录成功后将管理员会话保存到本地持久化和 Scoped 状态服务。
- 页面左上角始终展示应用图标和应用名称。

#### 1.4.2 配置保存交互

- 管理员修改基础配置后点击“保存配置”立即落盘。
- 保存成功后刷新首页摘要与微信示例页状态。
- 若关键字段缺失，则提示错误并阻止启动监听。

#### 1.4.3 微信绑定交互

- 点击“绑定微信”后调用二维码接口。
- 若已有未过期会话，则优先复用；点击“刷新二维码”强制重新获取。
- 页面展示二维码图片、扫码说明、会话 Key 与开始时间。
- 后台轮询扫码结果，成功后自动更新绑定状态与基础配置字段。
- 若二维码过期，则界面提示“二维码已过期”，并保留刷新入口。

#### 1.4.4 监听交互

- 点击“启动监听”时，系统先校验绑定状态和关键配置。
- 启动成功后运行后台长轮询服务并写入运行时间。
- 点击“停止监听”时取消轮询并写入停止时间。
- 若轮询异常但可恢复，则自动重试并写日志；若不可恢复，则状态切换为错误。

#### 1.4.5 测试推送交互

- 管理员输入 `ExternalChatId`、`ContextToken` 和消息内容。
- 点击“发送测试消息”后调用 `sendmessage`。
- 发送结果显示在当前区域，并写入交互日志。
- 若输入不完整，则按钮不可用或给出参数提示。

#### 1.4.6 用户发消息回复交互

- 当长轮询收到文本消息时，系统生成一条入站消息记录。
- 若自动回复开启，则使用回复模板生成文本并调用 `sendmessage` 回发。
- 若消息缺少 `context_token`，则仅记录消息与失败原因，不执行回包。
- 页面自动刷新消息表和日志区域，让演示过程可见。

#### 1.4.7 边界与错误处理

- API 请求失败：展示错误提示、保留最近一次成功状态、写日志。
- JSON 文件损坏：页面进入错误态，并提供重置示例数据入口。
- 未登录访问受保护页面：显示 `no-permission` 并跳转登录页。
- 关闭自动回复后收到消息：仍要保存入站记录，并在回复状态列中明确标记“未回复”。
